# 第二 API 配置指南

## 📋 概述

第二 API 是一个**完全独立**的 API 配置系统，专门用于执行辅助任务（如弹幕生成、商店刷新、系统对话、猜谜、生图标签生成等），与主 API 完全分离，互不影响。

### 为什么需要第二 API？

- **分离任务**：主 API 专注于剧情生成，第二 API 处理辅助功能，各司其职
- **成本优化**：主 API 使用高质量模型（如 GPT-4），第二 API 使用便宜模型（如 GPT-3.5），大幅降低成本
- **性能优化**：辅助功能不会阻塞或影响主剧情生成，提升响应速度
- **灵活配置**：可以为不同任务使用不同的提示词预设，精细控制
- **动态控制**：通过 MVU 变量动态切换任务提示词，一个预设管理所有任务

### 第二 API 与主 API 的区别

| 特性           | 主 API             | 第二 API                       |
| -------------- | ------------------ | ------------------------------ |
| **用途**       | 剧情生成、角色对话 | 弹幕、商店、系统聊天等辅助功能 |
| **配置位置**   | 酒馆主界面         | 游戏设置面板 → API 配置        |
| **模型选择**   | 通常使用高质量模型 | 可使用便宜模型                 |
| **提示词管理** | 角色卡、世界书     | 专用预设 + EJS 动态控制        |
| **调用时机**   | 用户发送消息时     | 特定功能触发时（如刷新商店）   |
| **是否必需**   | 必需               | 可选（不配置则相关功能禁用）   |

## 🎯 核心特性详解

### 1. 独立 API 配置

第二 API 拥有完全独立的配置系统，与主 API 互不干扰：

#### 配置项说明

| 配置项           | 说明                    | 示例                        |
| ---------------- | ----------------------- | --------------------------- |
| **接口地址**     | API 的 URL 地址         | `https://api.openai.com/v1` |
| **API 密钥**     | 用于身份验证的密钥      | `sk-...`                    |
| **模型名称**     | 使用的 AI 模型          | `gpt-3.5-turbo`、`gpt-4`    |
| **预设名称**     | 使用的提示词预设        | `第二API专用预设`           |
| **流式输出**     | 是否启用流式返回        | 开启/关闭                   |
| **温度**         | 控制输出随机性（0-2）   | 0.7（默认）                 |
| **最大回复长度** | 限制单次回复的 token 数 | 不设限或具体数值            |
| **Top P**        | 核采样参数（0-1）       | 0.9（默认）                 |
| **Top K**        | Top-K 采样参数（0-200） | 40（默认）                  |

#### 配置位置

1. 打开游戏界面
2. 点击右上角设置按钮
3. 滚动到"API 配置"部分
4. 找到"第二 API"配置区域

### 2. 预设支持与动态开关（核心机制）

这是第二 API 最重要的特性！通过 **MVU 变量** 和 **EJS 语法**，实现一个预设管理所有任务。

#### 工作原理

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 用户触发功能（如刷新商店）                                │
├─────────────────────────────────────────────────────────────┤
│ 2. 系统保存当前预设名称                                      │
├─────────────────────────────────────────────────────────────┤
│ 3. 加载第二 API 指定的预设                                   │
├─────────────────────────────────────────────────────────────┤
│ 4. 设置任务控制变量（如 vn_task_shop = true）               │
│    - 重置所有任务开关为 false                                │
│    - 只开启当前任务对应的开关                                │
│    - 等待变量同步到聊天变量（nextTick）                      │
├─────────────────────────────────────────────────────────────┤
│ 5. 发送 API 请求                                             │
│    - 预设中的 EJS 根据 vn_task_* 变量决定使用哪个提示词     │
│    - 例如：<% if (getvar('vn_task_shop')) { %> 商店提示词   │
├─────────────────────────────────────────────────────────────┤
│ 6. 接收并处理返回结果                                        │
├─────────────────────────────────────────────────────────────┤
│ 7. 恢复任务控制变量到原始状态                                │
├─────────────────────────────────────────────────────────────┤
│ 8. 恢复原预设                                                │
└─────────────────────────────────────────────────────────────┘
```

#### MVU 变量控制开关

系统使用以下聊天变量来控制不同任务的提示词：

| 变量名             | 对应任务     | 何时为 true          |
| ------------------ | ------------ | -------------------- |
| `vn_task_danmaku`  | 弹幕生成     | 调用弹幕生成时       |
| `vn_task_imageGen` | 生图标签生成 | 调用生图标签生成时   |
| `vn_task_shop`     | 商店物品生成 | 刷新商店时           |
| `vn_task_riddle`   | 谜题 AI 回复 | 猜谜游戏中 AI 回复时 |
| `vn_task_system`   | 系统聊天     | 系统联系人对话时     |

**重要**：这些变量由系统自动管理，你**不需要手动设置**，只需要在预设中使用 EJS 语法读取它们。

#### 预设中的 EJS 示例

```ejs
<%# 这是注释，不会被输出 %>

<%# 弹幕生成任务 %>
<% if (getvar('vn_task_danmaku')) { %>
你是一个弹幕生成器。根据以下内容生成 3-5 条简短有趣的弹幕评论（每行一条）：
<% } %>

<%# 商店物品生成任务 %>
<% if (getvar('vn_task_shop')) { %>
你是一个末日风格小店的商品生成器。
输出格式：商品名|效果描述|价格（整数）
输出 4 条，价格在 20–150 之间。
<% } %>

<%# 生图标签生成任务 %>
<% if (getvar('vn_task_imageGen')) { %>
根据场景描述生成适合的图片标签（英文，逗号分隔，不超过50词）。
<% } %>

<%# 弹幕+生图合并任务 %>
<% if (getvar('vn_task_danmaku') && getvar('vn_task_imageGen')) { %>
请完成两个任务：
1. 生成 3-5 条弹幕（每行一条）
2. 生成生图标签（最后一行，格式：IMAGE_TAG: 标签内容）
<% } %>

<%# 谜题 AI 回复任务 %>
<% if (getvar('vn_task_riddle')) { %>
你正在和用户玩猜谜游戏。用户会给你提示，你需要根据提示猜测一个词（谜底）。
你只能回复你的猜测或请求更多提示，不要重复用户的提示。
若你猜中了谜底，在回复中自然地说出答案即可。
<% } %>

<%# 系统聊天任务 %>
<% if (getvar('vn_task_system')) { %>
作为系统助手与用户对话。如果有剧情参考（通过 injects 注入），请结合参考内容回复。
<% } %>
```

### 3. 弹幕和生图合并调用

为了提高效率和减少 API 调用次数，弹幕和生图可以合并为一次调用。

#### 触发条件

同时满足以下条件时，会合并调用：

1. 弹幕功能已启用（`danmakuEnabled = true`）
2. 弹幕使用第二 API（`apiTaskDanmaku = 'second'`）
3. 生图功能已启用（`imageGenEnabled = true`）
4. 生图标签使用第二 API（`apiTaskImageTag = 'second'`）
5. 检测到消息中有场景变化或 CG 场景

#### 返回格式

预设需要返回包含弹幕和生图标签的文本，系统会自动解析：

**方式 1：使用 IMAGE_TAG: 标记**

```
这是第一条弹幕
这是第二条弹幕
这是第三条弹幕
IMAGE_TAG: background, post-apocalyptic, wasteland, ruins, newspaper style
```

**方式 2：使用中文标记**

```
这是第一条弹幕
这是第二条弹幕
这是第三条弹幕
生图标签: background, post-apocalyptic, wasteland, ruins
```

**方式 3：最后一行作为标签（无标记）**

```
这是第一条弹幕
这是第二条弹幕
这是第三条弹幕
background, post-apocalyptic, wasteland, ruins
```

系统会按以下优先级解析：

1. 查找 `IMAGE_TAG:` 或 `生图标签:` 标记
2. 如果没有标记，最后一行作为生图标签，其余作为弹幕

### 4. 系统聊天剧情引用（@ 功能）

系统聊天支持引用历史剧情楼层作为上下文，让 AI 能够结合剧情内容回复。

#### 使用方法

1. 在系统聊天界面，点击输入框旁的 **@ 按钮**
2. 在弹出的楼层选择器中输入：
   - **单楼层**：如 `3`（引用第 3 楼）
   - **楼层区间**：如 `1-5`（引用第 1 到 5 楼）
3. 点击"确认"，会显示引用标签（如"楼层 3（1 条）"）
4. 输入你的消息并发送
5. AI 会收到引用的剧情内容作为上下文

#### 技术实现

引用内容通过 **injects** 机制注入到提示词中：

```typescript
// 提示词结构
[
  { role: 'system', content: '系统提示词' },
  { role: 'user', content: '历史消息1' },
  { role: 'assistant', content: 'AI回复1' },
  // ↓ 这里注入引用内容（depth: 2）
  { role: 'user', content: '[剧情参考]\n楼层3的内容...' },
  { role: 'assistant', content: '好的，我已了解相关剧情内容。' },
  // ↓ 最后是用户当前输入（depth: 1）
  { role: 'user', content: '关于这个剧情，你怎么看？' }
]
```

**depth 参数说明**：

- `depth: 2` 表示注入到倒数第 2 层（用户输入之前）
- `depth: 1` 表示注入到倒数第 1 层（最后）

#### 注意事项

- 引用过多楼层可能超过 token 限制，建议不超过 10 楼
- 引用区间时，系统会获取该区间内所有消息
- 引用内容会自动去除 HTML 标签，只保留纯文本

### 5. API 任务分配

你可以为每个任务指定使用哪个 API，提供最大的灵活性。

#### 可配置的任务

| 任务         | 说明                     | 默认值   |
| ------------ | ------------------------ | -------- |
| **弹幕生成** | 生成弹幕文案             | 第二 API |
| **生图 Tag** | 根据场景生成图片标签     | 第二 API |
| **变量更新** | 更新游戏变量（预留功能） | 主 API   |

#### 配置选项

每个任务有三个选项：

- **主 API**：使用酒馆主界面配置的 API
- **第二 API**：使用第二 API 配置
- **禁用**：不执行该任务

#### 配置位置

1. 打开设置面板
2. 找到"第二 API"配置区域
3. 点击"API 任务分配与生成历史"按钮
4. 在弹出的面板中配置每个任务

## 📊 API 任务分配详解

### 什么是 API 任务分配？

API 任务分配允许你为每个具体任务选择使用哪个 API，提供最大的灵活性。

### 配置选项说明

每个任务有三个选项：

#### 1. 主 API

使用酒馆主界面配置的 API（通常是你的主要 AI 服务）。

**优点**：

- 使用高质量模型，生成效果更好
- 无需额外配置

**缺点**：

- 成本较高
- 可能影响主剧情生成的响应速度

**适用场景**：

- 对生成质量要求极高的任务
- 预算充足
- 不在意响应速度

#### 2. 第二 API

使用第二 API 配置（独立的 API 服务）。

**优点**：

- 成本低（可以使用便宜模型）
- 不影响主剧情生成
- 可以使用专门的提示词预设

**缺点**：

- 需要额外配置
- 生成质量可能略低于主 API（取决于模型选择）

**适用场景**：

- 辅助功能（弹幕、商店等）
- 成本敏感
- 需要高并发（主 API 和第二 API 可以同时工作）

#### 3. 禁用

完全不执行该任务。

**适用场景**：

- 不需要该功能
- 临时关闭某个功能
- 调试时排除某个功能的影响

### 配置建议

#### 推荐配置 1：成本优先

适合预算有限的用户：

| 任务     | 配置     | 原因                           |
| -------- | -------- | ------------------------------ |
| 弹幕生成 | 第二 API | 弹幕是辅助功能，GPT-3.5 足够   |
| 生图 Tag | 第二 API | 标签生成简单，不需要高质量模型 |
| 变量更新 | 禁用     | 暂未实现，无需配置             |

**预计成本**：每次剧情生成约 0.01-0.02 美元（假设主 API 使用 GPT-4，第二 API 使用 GPT-3.5）

#### 推荐配置 2：质量优先

适合预算充足、追求最佳效果的用户：

| 任务     | 配置   | 原因                        |
| -------- | ------ | --------------------------- |
| 弹幕生成 | 主 API | 使用 GPT-4 生成更有趣的弹幕 |
| 生图 Tag | 主 API | 使用 GPT-4 生成更准确的标签 |
| 变量更新 | 主 API | 预留                        |

**预计成本**：每次剧情生成约 0.05-0.08 美元（全部使用 GPT-4）

#### 推荐配置 3：混合模式

平衡成本和质量：

| 任务     | 配置     | 原因                             |
| -------- | -------- | -------------------------------- |
| 弹幕生成 | 第二 API | 弹幕对质量要求不高               |
| 生图 Tag | 主 API   | 生图标签影响图片质量，使用主 API |
| 变量更新 | 禁用     | 暂未实现                         |

**预计成本**：每次剧情生成约 0.03-0.04 美元

## 🐛 故障排除

### 问题 1：连接失败

**症状**：点击"测试连接"后显示"连接失败"

**可能原因**：

1. 接口地址错误
2. API 密钥错误或已过期
3. 网络问题（无法访问该地址）
4. API 服务商问题（服务器宕机）

**解决方法**：

**步骤 1**：检查接口地址

- 确认地址拼写正确
- 确认末尾没有多余的斜杠 `/`
- 确认使用 HTTPS 协议
- 尝试在浏览器中访问该地址（应该返回 404 或 API 文档页面）

**步骤 2**：检查 API 密钥

- 确认密钥完整（通常以 `sk-` 开头）
- 确认密钥未过期
- 尝试在 API 提供商的控制台中重新生成密钥

**步骤 3**：检查网络

- 确认网络连接正常
- 如果使用代理，确认代理配置正确
- 尝试使用其他网络（如手机热点）

**步骤 4**：检查 API 状态

- 访问 API 提供商的状态页面
- 查看是否有服务中断公告
- 尝试稍后再试

### 问题 2：模型列表获取失败

**症状**：点击"拉取模型"后显示"获取模型列表失败"

**可能原因**：

1. API 不支持 `/models` 接口
2. 密钥权限不足
3. 网络超时

**解决方法**：

**方法 1**：手动输入模型名称

- 不使用"拉取模型"功能
- 直接在下拉框中输入模型名称（如 `gpt-3.5-turbo`）
- 参考 API 提供商的文档获取可用模型列表

**方法 2**：检查密钥权限

- 确认密钥有权限访问 `/models` 接口
- 在 API 提供商的控制台中检查密钥权限设置

**方法 3**：增加超时时间

- 如果网络较慢，可能需要等待更长时间
- 目前超时时间为 8 秒，无法修改

### 问题 3：请求失败或超时

**症状**：使用功能时提示"请求失败"或长时间无响应

**可能原因**：

1. 网络问题
2. API 超时（超过 30 秒）
3. 模型不可用或已下线
4. 请求频率过高，被限流

**解决方法**：

**步骤 1**：检查网络连接

- 确认网络稳定
- 尝试刷新页面

**步骤 2**：检查模型可用性

- 确认所选模型仍然可用
- 尝试更换其他模型

**步骤 3**：等待重试

- 系统会自动重试 2 次
- 如果仍然失败，稍后再试

**步骤 4**：检查请求频率

- 如果短时间内频繁请求，可能被限流
- 等待几分钟后再试
- 考虑升级 API 套餐以提高限流阈值

### 问题 4：预设加载失败

**症状**：选择预设后提示"加载预设失败"或预设列表为空

**可能原因**：

1. 预设不存在
2. 预设名称错误
3. 预设文件损坏

**解决方法**：

**步骤 1**：刷新预设列表

- 点击预设下拉框旁的"刷新"按钮
- 查看预设是否出现在列表中

**步骤 2**：检查预设是否存在

- 在酒馆主界面中打开预设管理
- 确认预设确实存在
- 确认预设名称拼写正确

**步骤 3**：重新创建预设

- 如果预设损坏，删除并重新创建
- 参考本文档中的预设示例

**步骤 4**：选择"不使用预设"

- 如果预设问题无法解决，可以暂时选择"不使用预设"
- 使用默认参数进行测试

### 问题 5：弹幕或生图不显示

**症状**：配置正确，但弹幕或生图不显示

**可能原因**：

1. 功能未启用
2. 任务分配配置错误
3. 第二 API 未配置
4. 消息中没有触发条件

**解决方法**：

**步骤 1**：检查功能开关

- 弹幕：确认"启用弹幕"开关已开启
- 生图：确认"生图总开关"已开启，且至少开启了"背景生成"或"CG 生成"

**步骤 2**：检查任务分配

- 打开"API 任务分配与生成历史"面板
- 确认"弹幕生成"和"生图 Tag"配置为"第二 API"
- 如果配置为"禁用"，功能不会执行

**步骤 3**：检查第二 API 配置

- 确认第二 API 已正确配置
- 点击"测试连接"验证配置

**步骤 4**：检查触发条件

- 弹幕：AI 生成完成后自动触发
- 生图：需要消息中包含场景标记（如 `[[character||场景：xxx]]`）

**步骤 5**：查看控制台

- 按 F12 打开浏览器控制台
- 查看是否有错误信息
- 搜索 `[Danmaku]` 或 `[ImageGen]` 相关日志

### 问题 6：生成内容不符合预期

**症状**：生成的弹幕、商品等内容质量差或格式错误

**可能原因**：

1. 预设提示词不够清晰
2. 模型质量不够
3. 温度参数设置不当
4. 最大回复长度限制过小

**解决方法**：

**步骤 1**：优化预设提示词

- 使用更清晰、具体的指令
- 提供示例输出
- 明确格式要求

**步骤 2**：更换模型

- 尝试使用更高质量的模型（如 GPT-4）
- 不同模型的表现差异很大

**步骤 3**：调整温度参数

- 如果输出过于随机：降低温度（如 0.5）
- 如果输出过于死板：提高温度（如 0.9）

**步骤 4**：增加最大回复长度

- 如果输出被截断，增加"最大回复长度"
- 建议值：弹幕 200、商店 300、系统聊天 1000

### 问题 7：系统聊天引用不生效

**症状**：点击 @ 按钮引用楼层后，AI 回复没有结合引用内容

**可能原因**：

1. 预设中没有处理引用内容的指令
2. 引用内容被截断（超过 token 限制）
3. 模型忽略了引用内容

**解决方法**：

**步骤 1**：优化预设提示词

### 步骤 3：填写基础配置

#### 3.1 接口地址（API URL）

在第一个输入框中填写 API 的接口地址：

**OpenAI 官方**：

```
https://api.openai.com/v1
```

**其他兼容 OpenAI 格式的服务**：

```
https://your-api-provider.com/v1
```

**注意事项**：

- 地址末尾不要有多余的斜杠 `/`
- 必须是 HTTPS 协议（安全连接）
- 确保地址可以访问（不被墙）

#### 3.2 API 密钥（API Key）

在第二个输入框中填写你的 API 密钥：

```
sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

**注意事项**：

- 密钥是敏感信息，不要分享给他人
- 输入框会自动隐藏密钥内容（显示为 ****）
- 如果密钥错误，测试连接会失败

#### 3.3 模型名称

有两种方式选择模型：

**方式 1：手动输入**

- 直接在下拉框中输入模型名称
- 常用模型：
  - `gpt-3.5-turbo`（便宜，适合辅助功能）
  - `gpt-4`（贵，质量高）
  - `gpt-4-turbo`（平衡）
  - `claude-3-sonnet`（如果使用 Claude）

**方式 2：自动拉取**

1. 填写好接口地址和密钥后
2. 点击"**拉取模型**"按钮
3. 系统会自动获取可用模型列表
4. 从下拉框中选择你想要的模型

**注意事项**：

- 如果拉取失败，说明 API 不支持 `/models` 接口，请手动输入
- 不同模型的价格和性能差异很大，根据需求选择

#### 3.4 使用的预设

选择第二 API 使用的提示词预设：

**选项说明**：

- **不使用预设（使用默认参数）**：不加载任何预设，使用系统默认配置
- **具体预设名称**：使用该预设的提示词配置

**如何选择预设**：

1. 点击下拉框
2. 从列表中选择你创建的第二 API 专用预设

- 预设需要在酒馆中预先创建（不是在游戏中创建）
- 预设中需要使用 EJS 语法根据 `vn_task_*` 变量控制提示词
- 选择预设后，API 请求仍使用上方配置的第二 API 地址和密钥

### 步骤 4：测试连接

填写完基础配置后，建议先测试连接：

1. 点击"**测试连接**"按钮
2. 等待几秒钟
3. 查看结果：
   - ✓ **连接成功，密钥可用**：配置正确，可以继续

**常见失败原因**：

- 接口地址错误（拼写错误、多余斜杠）
- API 密钥错误或已过期
- 网络问题（无法访问该地址）
- API 服务商问题（服务器宕机）

### 步骤 5：配置高级参数（可选）

如果需要精细控制生成效果，可以配置以下参数：

#### 5.1 流式输出

- **开启**：实时显示生成内容（逐字显示）
- **关闭**：等待全部生成完成后一次性显示

**建议**：关闭（第二 API 通常用于后台任务，不需要流式）

#### 5.2 最大回复长度（Max Tokens）

限制单次回复的最大 token 数：

- **不设限**：留空，由模型自行决定
- **具体数值**：如 `500`，限制最多生成 500 个 token

**建议**：

- 弹幕生成：`200`（弹幕很短）
- 商店物品：`300`（4 个商品）
- 系统聊天：`1000`（对话较长）
- 生图标签：`100`（标签很短）

#### 5.3 温度（Temperature）

控制输出的随机性和创意性：

- **0.0**：完全确定性，每次输出相同
- **0.7**：平衡（默认值）
- **1.0**：较高创意性
- **2.0**：极高随机性

**建议**：

- 弹幕生成：`0.9`（需要创意和多样性）
- 商店物品：`0.8`（需要一些随机性）
- 系统聊天：`0.7`（平衡）
- 生图标签：`0.5`（需要稳定性）

#### 5.4 Top P（核采样）

控制采样范围（0-1）：

- **0.9**：默认值，平衡
- **1.0**：考虑所有可能的词
- **0.5**：只考虑概率最高的 50% 的词

**建议**：保持默认值 `0.9`

#### 5.5 Top K

控制采样的候选词数量（0-200）：

- **40**：默认值
- **0**：不限制
- **10**：只考虑概率最高的 10 个词

**建议**：保持默认值 `40`

### 步骤 6：配置 API 任务分配

决定每个任务使用哪个 API：

1. 在第二 API 配置区域底部，点击"**API 任务分配与生成历史**"按钮
2. 在弹出的面板中，为每个任务选择：
   - **主 API**：使用酒馆主界面配置的 API
   - **第二 API**：使用第二 API 配置
   - **禁用**：不执行该任务

**任务说明**：

| 任务         | 建议配置 | 原因                           |
| ------------ | -------- | ------------------------------ |
| **弹幕生成** | 第二 API | 弹幕是辅助功能，用便宜模型即可 |
| **生图 Tag** | 第二 API | 标签生成简单，不需要高质量模型 |
| **变量更新** | 主 API   | 预留功能，暂未实现             |

### 步骤 7：启用相关功能

配置完第二 API 后，还需要启用对应的功能：

#### 7.1 启用弹幕

1. 在设置面板中找到"**弹幕设置**"部分
2. 开启"**启用弹幕**"开关
3. 配置弹幕相关选项：
   - **弹幕速度**：控制弹幕移动速度
   - **循环弹幕**：弹幕消失后是否重新出现
   - **显示区域**：弹幕显示在屏幕的哪个区域（1/3 屏、半屏、全屏）

**注意**：弹幕生成时，系统会自动提取最新消息中 `<content>` 标签内的文本作为输入，无需额外配置。

#### 7.2 启用生图

1. 在设置面板中找到"**生图**"部分
2. 开启"**生图总开关**"
3. 选择生成类型：
   - **背景生成**：生成场景背景图
   - **CG 生成**：生成 CG 插图
4. 如果两者都开启，选择优先级：
   - **CG 优先**：同时检测到两者时，优先生成 CG
   - **背景优先**：同时检测到两者时，优先生成背景

**注意**：生图功能需要安装支持生图事件的外部插件，第二 API 只负责生成标签。

配置完成后，设置会**自动保存**，无需手动保存。

关闭设置面板后，配置立即生效。

1. 弹幕功能已启用（`danmakuEnabled = true`）

3. AI 生成完成（`GENERATION_ENDED` 事件触发）

```
1. AI 生成剧情回复完成
   ↓
2. 系统提取消息内容（<content> 标签内的文本）

   ↓
5. 接收返回结果（每行一条弹幕）
   ↓
6. 解析弹幕列表
   ↓
7. 以 200ms-3s 的随机间隔依次显示弹幕
```

```ejs
<% if (getvar('vn_task_danmaku')) { %>

- 每行一条弹幕
- 每条弹幕不超过 20 字

<% } %>
```

```
前方高能预警
心疼主角一秒
这波操作666
```

- **温度**：0.9（需要创意和多样性）

**输入内容说明**：

### 场景 2：生图标签生成

#### 功能说明

根据剧情内容自动生成图片标签，用于调用生图插件生成背景或 CG。

#### 触发条件

1. 生图功能已启用（`imageGenEnabled = true`）
2. 生图标签使用第二 API（`apiTaskImageTag = 'second'`）
3. 检测到消息中有场景变化或 CG 场景标记
4. AI 生成完成（`GENERATION_ENDED` 事件触发）

#### 场景检测规则

系统会自动检测消息中的特殊标记：

**背景场景标记**：

```
[[character||角色名：xxx||场景：废墟中的旧报社||台词：xxx]]
```

**CG 场景标记**：

```
[[character||角色名：xxx||CG场景：夕阳下的拥抱||台词：xxx]]
```

**检测逻辑**：

- 如果有 `CG场景：`，判定为生成 CG
- 如果有 `场景：`（但没有 `CG场景：`），判定为生成背景
- 如果两者都有，根据用户设置的优先级决定

#### 工作流程

```
1. AI 生成剧情回复完成
   ↓
2. 检测消息中的场景标记
   ↓
3. 提取 <content> 标签内的场景描述文本
   ↓
4. 调用第二 API（设置 vn_task_imageGen = true）
   ↓
5. 预设中的 EJS 输出生图标签生成提示词
   ↓
6. 接收返回结果（英文标签，逗号分隔）
   ↓
7. 将标签发送给生图插件
   ↓
8. 生图插件生成图片并返回
   ↓
9. 显示图片并添加到卡牌队列
```

#### 预设示例

```ejs
<% if (getvar('vn_task_imageGen')) { %>
根据以下场景描述，生成适合的图片标签（英文，逗号分隔）。

要求：
- 使用英文单词或短语
- 用逗号分隔
- 不超过 50 个词
- 包含：场景类型、环境、氛围、风格

示例输出：
background, post-apocalyptic, wasteland, ruins, abandoned building, newspaper office, dusty, dim lighting, cinematic, realistic
<% } %>
```

#### 返回格式

```
background, post-apocalyptic, wasteland, ruins, abandoned newspaper office, dusty, dim lighting, papers scattered, broken windows, cinematic, realistic, detailed
```

#### 配置建议

- **温度**：0.5（需要稳定性，避免过于随机）
- **最大回复长度**：100（标签很短）
- **模型选择**：GPT-3.5 即可（标签生成不需要高质量模型）

### 场景 6：猜谜游戏

#### 功能说明

#### 触发条件

1. 第二 API 已配置（`secondApiStatus = 'available'`）
2. 用户在猜谜模块中开始游戏
3. 用户提供提示后，AI 进行猜测

#### 工作流程

```
1. 用户输入谜底和第一条提示
   ↓
2. 系统记录谜底（归一化处理）
   ↓
3. 用户提供提示
   ↓
4. 调用第二 API（设置 vn_task_riddle = true）
   ↓
5. 预设中的 EJS 输出猜谜提示词
   ↓
6. AI 根据提示进行猜测
   ↓
7. 系统检查 AI 的回复是否包含谜底
   ↓
8. 如果包含谜底：
   - 游戏结束，计算奖励
   - 奖励 = 50 + 回合数 × 20
   ↓
9. 如果不包含谜底：
   - 继续下一回合
   - 用户提供更多提示
```

- 全角字符转半角
- 去除所有空格
- 转为小写

例如：

- `"苹 果"` → `"苹果"`
- `"APPLE"` → `"apple"`
- `"Ａｐｐｌｅ"` → `"apple"`

## 🐛 故障排除

### 问题 1：连接失败

**可能原因**：

- 接口地址错误
- API 密钥错误
- 网络问题

**解决方法**：

1. 检查接口地址是否正确（注意末尾不要有多余的斜杠）
2. 检查 API 密钥是否正确
3. 尝试在浏览器中直接访问接口地址

### 问题 2：模型列表获取失败

**可能原因**：

- API 不支持 `/models` 接口
- 密钥权限不足

**解决方法**：

1. 手动输入模型名称
2. 检查 API 文档确认是否支持模型列表接口

### 问题 3：请求失败

**可能原因**：

- 网络问题
- API 超时
- 模型不可用

**解决方法**：

1. 检查网络连接
2. 检查 API 状态
3. 尝试更换模型
4. 系统会自动重试 2 次

### 问题 4：预设加载失败

**可能原因**：

- 预设不存在
- 预设名称错误

**解决方法**：

1. 点击"刷新"按钮更新预设列表
2. 检查预设是否存在于酒馆中
3. 选择"不使用预设"

### 附录 D：技术参数详解

#### 温度（Temperature）

控制输出的随机性和创意性：

| 值      | 效果                     | 适用场景             |
| ------- | ------------------------ | -------------------- |
| **0.0** | 完全确定性，每次输出相同 | 需要稳定输出的场景   |
| **0.3** | 非常保守，输出稳定       | 生图标签、格式化输出 |
| **0.5** | 较为保守，略有变化       | 商店物品、谜题回复   |
| **0.7** | 平衡（默认值）           | 系统聊天、一般对话   |
| **0.9** | 较高创意性，输出多样     | 弹幕生成、创意内容   |
| **1.2** | 高创意性，可能不稳定     | 实验性场景           |
| **2.0** | 极高随机性，输出不可预测 | 不推荐               |

#### Top P（核采样）

控制采样范围：

| 值      | 效果                            | 说明         |
| ------- | ------------------------------- | ------------ |
| **0.1** | 只考虑概率最高的 10% 的词       | 输出非常保守 |
| **0.5** | 只考虑概率最高的 50% 的词       | 较为保守     |
| **0.9** | 考虑概率最高的 90% 的词（默认） | 平衡         |
| **1.0** | 考虑所有可能的词                | 最大多样性   |

#### Top K

控制采样的候选词数量：

| 值      | 效果                             | 说明       |
| ------- | -------------------------------- | ---------- |
| **0**   | 不限制                           | 考虑所有词 |
| **10**  | 只考虑概率最高的 10 个词         | 非常保守   |
| **40**  | 只考虑概率最高的 40 个词（默认） | 平衡       |
| **100** | 只考虑概率最高的 100 个词        | 较高多样性 |

#### 最大回复长度（Max Tokens）

限制单次回复的 token 数：

| 任务          | 推荐值 | 说明                     |
| ------------- | ------ | ------------------------ |
| **弹幕生成**  | 200    | 3-5 条弹幕，每条约 20 字 |
| **生图标签**  | 100    | 标签很短，50 词以内      |
| **弹幕+生图** | 300    | 弹幕 + 标签              |
| **商店物品**  | 300    | 4 个商品                 |
| **系统聊天**  | 1000   | 对话可能较长             |
| **猜谜回复**  | 200    | 猜测很短                 |

**注意**：1 个中文字符约等于 2-3 个 tokens，1 个英文单词约等于 1-2 个 tokens。

### 附录 E：常见问题汇总

#### Q1：第二 API 和主 API 可以使用同一个密钥吗？

**A**：可以，但不推荐。使用独立密钥的好处：

- 便于分别统计使用量和成本
- 可以设置不同的使用限制
- 一个密钥出问题不影响另一个
- 便于管理和撤销

#### Q2：预设必须使用 EJS 语法吗？

**A**：是的。第二 API 的核心机制就是通过 EJS 根据 `vn_task_*` 变量动态切换提示词。如果不使用 EJS，所有任务会使用相同的提示词，失去灵活性。

#### Q3：可以为不同任务创建多个预设吗？

**A**：不需要。第二 API 的设计理念就是"一个预设管理所有任务"。通过 EJS 条件判断，一个预设可以包含所有任务的提示词，系统会根据任务类型自动选择。

#### Q4：第二 API 支持流式输出吗？

**A**：支持，但不推荐。第二 API 通常用于后台任务（如弹幕、商店），不需要实时显示生成过程。关闭流式输出可以提高稳定性。

#### Q5：为什么弹幕和生图要合并调用？

**A**：合并调用的好处：

- 减少 API 调用次数，降低成本
- 减少网络请求，提高响应速度
- 避免两次调用之间的延迟
- 更高效地利用 token

#### Q6：系统聊天的引用功能有什么限制？

**A**：主要限制：

- 引用内容会占用 token，可能超过模型限制
- 建议不要引用超过 10 楼
- 引用区间时注意内容长度
- 引用内容会自动去除 HTML 标签

#### Q7：如何查看第二 API 的使用情况？

**A**：目前游戏内没有统计功能，建议：

- 在 API 提供商的控制台查看使用量
- 使用独立密钥便于统计
- 定期检查账单

#### Q8：第二 API 配置会保存在哪里？

**A**：配置保存在浏览器的本地存储中（localStorage），与游戏存档分离。清除浏览器缓存会丢失配置，建议记录配置信息。

#### Q9：可以在手机上使用第二 API 吗？

**A**：可以，但需要注意：

- 手机浏览器的本地存储可能不稳定
- 输入配置信息较为不便
- 建议在电脑上配置好后，通过云同步使用

#### Q10：第二 API 支持哪些 API 提供商？

**A**：支持所有兼容 OpenAI API 格式的提供商，包括：

- OpenAI 官方
- Azure OpenAI
- Anthropic Claude（通过兼容层）
- 国内各大模型提供商（如智谱、百川等）
- 自建 API 服务

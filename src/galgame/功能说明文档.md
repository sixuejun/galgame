## 一、整体架构

### 技术栈

- **Vue 3** + **Pinia** 状态管理
- **Tailwind CSS v4** + 自定义 CSS 变量（`theme.css`）
- **流式楼层界面**：基于 `mountStreamingMessages` 挂载，每个 AI 消息楼层独立 iframe Vue 实例
- 持久化：设置存于 `script` 变量，游戏数据存于 `chat` 变量

### 视觉风格

- 复古做旧报纸质感：泛黄纸张、褪色油墨、折痕污渍
- 色调：灰暗土黄 `#d4c5a0`、灰黑 `#2a2420`、锈色 `#8b4513`
- 16:9 固定比例视口，报纸装饰元素（EXTRA EDITION / 末日旧闻 / 第壹百零七期）

### 图层结构（从下到上）

1. **背景层**：渐变 + 报纸标题水印
2. **生图背景层**：第二API生成的背景图片
3. **生图CG层**：第二API生成的CG图片
4. **弹幕层**：screen-space，不受场景抖动影响
5. **暗角层**：径向渐变暗角
6. **噪点层**：SVG fractalNoise 纹理
7. **扇形卡牌队列**（z-index 30）：左侧中间位置，悬停展开
8. **UI 层**（z-index 20）：快捷菜单 + 对话框
9. **选项层**：玩家选择面板
10. **覆盖面板**（z-index 50）：设置/历史/角色/玩法
8. **模块面板**（z-index 55）：各玩法模块 overlay
9. **系统通知**（z-index 9999）：Toast 弹窗，永远在最上层

## 四、经济系统 `EconomyService`

- **统一走账**：所有金币变更必须通过 `changeGold(amount, moduleId, reason)`
- **交易日志**：最多保留 50 条记录（模块ID / 原因 / 数额 / 时间戳）
- **背包系统**：`addInventoryItem` 支持堆叠
- **抽成公式**：`calcCommission(gold, workshopLevel, basePrice, threshold=500)`
  - 金币 ≤ 阈值时仅收取基价
  - 超过阈值按工坊等级 0.5%/级收取，封顶 5%
- **股票手续费**：`calcStockFee(workshopLevel)` = workshopLevel × 5，封顶 50

## 五、玩法模块

### 5.1 背包 `InventoryModule`

- 物品列表（名称 / 效果 / 数量）
- 总物品数统计

### 5.2 金币记录 `GoldLogModule`

- 交易日志列表（模块标签 / 原因 / 时间 / 金额增减）
- 当前余额显示

### 5.3 商店 `ShopModule`

- **解锁条件**：需配置第二 API
- 付费刷新（基价 50G + 抽成），携带 generationId 防并发
- 购买即时扣款、物品即时入背包
- 刷新中禁用按钮

### 5.4 工坊 & 交易 `WorkshopModule`（合并工坊 + 股市）

#### 工坊生产

- 工坊等级 1-10，每级 +10% 产出加成
- 选择已解锁角色开始生产
- 按真实时间每秒计算即时增加（离线不算）
- **快速暂停/继续**：暂停时累计已产出，继续时从暂停点继续
- 停止结算：一次性计算差值入账
- 升级消耗：等级 × 200G

#### 风险与机遇（股市）

- **与生产并行**：经营股票不打断生产
- **现金同步**：可支配现金即玩家金币（允许生产收入实时增加可用金额）
- **价格模拟**：离散随机游走 + 轻微均值回归 + 交易冲击
  - 3 秒一次自动变化，交易后立即变化
  - 暂停时折线不变化、无法交易
- **操作**：
  - 买入（+）：按当前价 + 手续费 扣除金币，持仓 +1
  - 卖出（-）：持仓 -1，按当前价 - 手续费 返还金币
  - 可连点快速操作
- **折线图**：
  - 浅色背景 + 水平/垂直网格线 + 价格标签
  - 涨跌用绿色/红色箭头标注
  - 渐变填充（涨绿 / 跌红）
  - 退出（返回）时自动重置
  - 使用 ResizeObserver + nextTick 确保渲染
- **刷新**：随机生成不同初始价格开局
- **暂停/继续**：图标按钮控制
- **退出**：自动平仓卖出全部持仓
- **预计收益**：实时计算（持仓价值 - 手续费 - 已投入）

### 5.5 思绪整理 `Puzzle2048Module`（2048 合成小游戏）

- **点开即玩**：点击卡片自动扣费并开始
- **费用**：金币 ≤ 500 免费，超过按抽成公式计算
- **动态棋盘**：
  - 默认 4×4
  - 4 级工坊解锁 6×6
  - 8 级工坊解锁 8×8
  - 有多个可选大小时显示选择器
- 标准 2048 规则，方块封顶 2048
- 键盘（WASD / 方向键）+ 触屏滑动操作
- **收益**：方块数字之和 × 2048 倍率
  - 0 个 2048: ×1.0
  - 1 个 2048: ×1.5
  - 2 个 2048: ×2.2
  - 3+ 个 2048: ×3.0
- 保存退出：保留棋盘、不结算
- 结算退出：清空棋盘、收益入账

### 5.6 情报交换 `RiddleModule`（AI 对话猜谜）

- **解锁条件**：需配置第二 API
- 用户输入谜底（本地校验，不进入 prompt）
- 用户输入第一条提示开始游戏
- 对话式猜谜，AI 逐轮回复
- **谜底屏蔽**：用户输入包含谜底时阻止发送
- **胜利条件**：AI 回复匹配到谜底
- **奖励**：50 + 轮数 × 20 金币
- 保留上次战绩记录

## 八、长程记忆系统

### 8.1 核心功能

**自动总结机制**：

- **小总结**：每 N 轮（默认 5 轮）自动生成，记录最近对话的关键信息（200字以内）
- **大总结**：当小总结累积到阈值（默认 6 个）时，自动提炼为长期记忆（300字以内）
- **手动触发**：支持手动触发大总结生成
- **编辑/删除**：可以编辑和删除任何总结内容

**数据持久化**：

- 所有总结数据存储到**聊天变量**，每个存档独立
- 大总结自动清理（最多保留 10 个）
- 小总结在生成大总结后自动清空

### 8.2 配置项

| 配置项                 | 说明                     | 默认值 |
| ---------------------- | ------------------------ | ------ |
| `summarySmallInterval` | 小总结间隔（轮数）       | 5      |
| `summaryBigThreshold`  | 大总结阈值（小总结数量） | 6      |
| `summaryAutoEnabled`   | 是否自动生成总结         | true   |
| `apiTaskSummary`       | 使用哪个API生成          | second |

### 8.3 工作流程

```
轮次1-5   → 生成小总结1
轮次6-10  → 生成小总结2
轮次11-15 → 生成小总结3
轮次16-20 → 生成小总结4
轮次21-25 → 生成小总结5
轮次26-30 → 生成小总结6
          → 触发大总结（包含小总结1-6）
          → 清空小总结，继续累积
```

### 8.4 UI 界面

**历史面板 - 总结 Tab**：

- Tab 切换（对话记录 / 总结）
- 显示所有大总结和小总结
- 支持展开/收起小总结列表
- 点击编辑按钮可修改总结内容
- 显示总结配置和当前轮数
- 手动触发大总结按钮

**配置区域**：

- 小总结间隔调整（1-50 轮）
- 大总结阈值调整（2-20 个）
- 自动总结开关

### 8.5 数据存储

```typescript
// 聊天变量（每个存档独立）
{
  vn_summary_small: SmallSummary[],      // 小总结列表
  vn_summary_big: BigSummary[],          // 大总结列表
  vn_summary_current_round: number,      // 当前轮数
}

// 小总结结构
interface SmallSummary {
  id: string;
  content: string;
  rounds: string;        // 如 "1-5"
  timestamp: number;
}

// 大总结结构
interface BigSummary {
  id: string;
  content: string;
  rounds: string;        // 如 "1-30"
  timestamp: number;
  smallCount: number;    // 包含的小总结数量
}
```

### 8.6 使用建议

**短对话**（<20轮）：

- 小总结间隔：5 轮
- 大总结阈值：4 个

**中等对话**（20-50轮）：

- 小总结间隔：5 轮
- 大总结阈值：6 个

**长对话**（>50轮）：

- 小总结间隔：10 轮
- 大总结阈值：5 个

## 九、第二API增强

### 9.1 API任务分配系统

**灵活分配**：每个功能可独立选择使用主API、第二API或禁用

| 任务     | 说明                 | 默认API |
| -------- | -------------------- | ------- |
| 弹幕生成 | 生成弹幕文案         | 第二API |
| 总结生成 | 生成小总结和大总结   | 第二API |
| 生图Tag  | 根据场景生成图片标签 | 第二API |
| 变量更新 | 更新游戏变量（预留） | 主API   |

**配置入口**：

- 设置面板 → API 配置 → "API 任务分配与生成历史"按钮

### 9.2 生图Tag自动生成

**核心功能**：

- 使用第二API根据场景描述生成图片标签
- 智能降级：第二API不可用时自动使用原始场景描述
- 内容追踪：生成的tag自动插入到消息末尾（HTML注释格式）

**工作流程**：

```
消息生成 → 提取场景描述 → 第二API生成tag → 发送给生图插件
```

**示例**：

```
场景描述: "废墟中的少年在收集旧报纸"
↓
第二API生成tag: "post-apocalyptic, young boy, collecting newspapers, ruins, wasteland"
↓
发送给生图插件生成图片
```

### 9.3 内容插入格式

第二API生成的内容会以HTML注释格式插入到消息末尾：

```html
<!-- 第二API生成(summary): 这是一个总结内容... -->
<!-- 第二API生成(imageTag): post-apocalyptic, wasteland, newspaper style -->
<!-- 第二API生成(danmaku): 弹幕1\n弹幕2\n弹幕3 -->
```

**特点**：

- 不影响前端显示
- 可在消息源码中查看
- 用于调试和追踪

### 9.4 调试信息

Console日志：

[Summary] Triggering small summary    // 触发小总结
[Summary] Triggering big summary      // 触发大总结

## 十、世界书管理

### 10.0 使用步骤与运作原理

**入口**：设置面板 → API 配置区域 → 点击 **「世界书管理」** 按钮。

**数据来源**：面板会拉取**当前对话下所有用到的世界书**，包括：当前角色卡绑定的世界书（主世界书 + 附加世界书）、当前聊天绑定的世界书、以及全局世界书。若有多个世界书，会按世界书分组显示。

**如何控制「哪个条目发送、哪个不发送」**：

- **发送 = 启用，不发送 = 禁用**。酒馆在组 prompt 时只会把 **已启用**（`enabled: true`）的世界书条目注入到上下文中；禁用的条目不会发给任何 API。
- 在面板里每个条目右侧有一个按钮，显示 **「已启用」** 或 **「已禁用」**，点击即可在两种状态间切换。切换后通过酒馆助手的 `updateWorldbookWith` 写回世界书，之后请求主 API / 第二 API 时都会按新的启用状态生效。
- 可选地，你还可以为每个条目设置：
  - **发送给**：该条目若启用，是发给 **主 API**、**第二 API** 还是 **两者都发送**（仅影响本前端/脚本侧对「发给谁」的约定；实际是否注入仍由酒馆根据条目的 `enabled` 决定）。
  - **关联功能**：把条目标记为与「弹幕」或「生图」相关，便于和下面的「自动控制」配合。
  - **自动控制**：勾选后，当你在设置里打开/关闭「弹幕」或「生图」时，本前端会自动把**同一世界书内**、关联了该功能且勾了自动控制的条目批量设为启用或禁用，无需你每次手动点。

**运作流程简述**：

1. 打开世界书管理 → 点击刷新（或首次打开自动加载）→ 拉取当前角色/聊天/全局世界书的所有条目。
2. 对每个条目：用「已启用/已禁用」决定**是否参与发送**；用「发送给」和「关联功能 + 自动控制」做细粒度与自动化管理。
3. 启用状态与 `extra`（如 targetApi、linkedFeature、autoControl）会通过 `updateWorldbookWith` 写回酒馆世界书，后续对话请求都会按最新状态使用这些条目。

### 10.1 核心功能

**条目增强**：

- **启用/禁用**：手动控制每个世界书条目是否启用
- **API分配**：每个条目可选择发送给主API、第二API或两者
- **自动控制**：根据功能开关（弹幕、生图、总结）自动启用/禁用关联条目
- **关联功能**：将条目与特定功能关联，实现智能管理

### 10.2 条目增强字段

```typescript
interface WorldbookEntryEnhanced {
  enabled: boolean;              // 是否启用
  targetApi: 'main' | 'second' | 'both';  // 发送给哪个API
  autoControl: boolean;          // 是否自动控制
  linkedFeature?: 'danmaku' | 'imageGen' | 'summary';  // 关联的功能
}
```

### 10.3 自动控制逻辑

**触发条件**：

- 当 `danmakuEnabled` 开启时，自动启用 `linkedFeature: 'danmaku'` 的条目
- 当 `imageGenEnabled` 开启时，自动启用 `linkedFeature: 'imageGen'` 的条目
- 当 `summaryAutoEnabled` 开启时，自动启用 `linkedFeature: 'summary'` 的条目

**使用场景**：

```
用户开启弹幕功能
  → 自动启用所有 linkedFeature='danmaku' 的世界书条目
  → 这些条目包含弹幕生成的提示词
  → 第二API根据这些提示词生成弹幕

用户关闭弹幕功能
  → 自动禁用相关条目
  → 节省token消耗
```

### 10.4 UI 界面

**世界书管理面板**：

- 显示所有世界书条目
- 每个条目可设置：
  - 启用状态（开/关）
  - API分配（主API / 第二API / 两者）
  - 关联功能（弹幕 / 生图 / 总结 / 无）
  - 自动控制开关

## 十一、生图系统与扇形卡牌队列

### 11.1 概述

生图系统通过前端助手事件与外部插件通信，支持在主API回复后自动调用第二API生成图片tag，并通过插件生成背景图或CG图。

### 11.2 生图开关逻辑

| 开关状态 | 行为 |
|---------|------|
| 仅背景 | 只生成背景图片，默认生成的图片作为背景 |
| 仅CG | 只生成CG图片，默认生成的图片作为CG |
| 两者都开 | 根据消息内容判断生成类型，开启扇形卡牌队列 |
| 都关闭 | 不调用生图 |

### 11.3 插入位置判断规则

当同时开启背景和CG生图时，系统会根据消息内容判断：

1. **优先判定CG**：如果消息中出现CG场景块
   ```
   [[character||角色名：程北极||CG场景：告白||台词：我喜欢你]]
   ```

2. **判定背景**：如果出现场景变化（但不是CG场景）
   ```
   [[character||角色名：程北极||场景：家中||xxxx]]
   [[character||角色名：程北极||场景：学校||xxxx]]
   ```

3. **两者都存在**：使用用户设置的优先级（CG优先/背景优先）

### 11.4 第二API生图流程

1. **主API回复完成**：触发 `GENERATION_ENDED` 事件
2. **判断生图类型**：根据消息内容和开关状态判断
3. **调用第二API**：生成图片tag（如果启用）
   ```typescript
   const tagPrompt = `根据以下场景描述，生成适合的图片tag（英文，逗号分隔，不超过50词）：\n${context}`;
   ```
4. **发送生图请求**：通过 `eventEmit` 发送到外部插件
5. **写入消息末尾**：将第二API生成的tag作为HTML注释写入
   ```html
   <!-- 第二API生图tag: post-apocalyptic, wasteland, newspaper style -->
   ```

### 11.5 扇形卡牌队列 (Image Deck)

#### 显示条件
- 同时打开背景和CG生图开关
- 至少有1张图片在队列中

#### 交互方式
- **默认收起**：左侧中间位置显示叠卡角，显示队列数量
- **悬停展开**：变成扇形轮盘，按顺序排列图片
- **点击卡牌**：立即切换舞台显示为该卡牌对应图片

#### 队列管理
- 最多保存10张图片
- 超过10张时自动移除最旧的
- 可在设置面板中清空队列

#### 卡牌样式
- **背景卡牌**：灰褐色边框，标记"BG"
- **CG卡牌**：锈色边框，标记"CG"
- 悬停时放大并高亮边框

### 11.6 配置选项

在设置面板的"生图"部分：

- **生图总开关**：控制整个生图功能
- **背景生成**：开启背景图生成
- **CG生成**：开启CG图生成
- **两者都存在时的优先级**：CG优先 / 背景优先
- **卡牌队列**：显示当前队列数量，提供清空按钮

### 11.7 技术实现

#### Store 状态
```typescript
const stageBackgroundImage = ref<string | null>(null);
const stageCgImage = ref<string | null>(null);
const imageCardQueue = ref<ImageCard[]>([]);

interface ImageCard {
  id: string;
  imageData: string; // base64
  type: 'background' | 'cg';
  timestamp: number;
}
```

#### 事件通信
```typescript

  id: requestId,
  prompt: prompt,
  width: null,
  height: null,
});

// 监听响应
eventOn(ImageGenEventType.GENERATE_IMAGE_RESPONSE, (responseData) => {
  // 添加到卡牌队列
  // 更新舞台显示
});
```
